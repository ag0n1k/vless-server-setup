## Цель проекта

Спрятать WireGuard-трафик внутри легитимного DTLS-медиапотока **группового звонка ВКонтакте**, заворачивая его через публичные TURN-серверы VK так, чтобы DPI видел "обычный VK-call" вместо VPN.

## Архитектура (версия v1.0.6 / v1.1.0)

### E2E-поток пакета

```
[Android app: AllowedIPs трафик]
   │
   ▼  (внутри WireGuardGoBackend)
[loopback UDP 127.0.0.1:9000]   ← виртуальный кабель между WG-движком и Go-ядром
   │
   ▼  Dispatcher (round-robin по 24-72 воркерам)
[Worker #N: pion/turn/v5 клиент]
   │   STUN/TURN Allocate с VK-кредами (User/Pass из VK call)
   ▼
[VK TURN relay] ← внешний IP:порт VK CDN (calls.okcdn.ru/прочие)
   │   (TURN ChannelData — обёртка для UDP через VK)
   ▼
[connutil.AsyncPacketPipe] ← in-process pipe
   │
   ▼  pion/dtls/v3 client (TLS 1.2 ECDHE-ECDSA-AES128-GCM, SNI=calls.okcdn.ru, ConnectionID)
[DTLS handshake → wdtt-server на VPS]
   │
   ▼  payload: либо "GETCONF:port|deviceID|password", либо raw WG UDP
[VPS: wdtt-server слушает 56000/udp]
   │   парсит WG-payload, форвардит в локальный wg0 на 56001/udp
   ▼
[wg-quick wdtt0 интерфейс] → NAT/MASQUERADE → интернет
```

### Ключевые модули Go-клиента (`go_client/`)

#### `main.go` — bootstrap
- Парсит флаги (`-peer`, `-vk` хеши, `-n` воркеров, `-tcp/-udp`, `-device-id`, `-password`, `-captcha-mode`)
- stdin для управляющих команд от Kotlin-обёртки: `PAUSE`/`RESUME` (Doze-mode), `STOP`, `CAPTCHA_RESULT|<token>`
- Локальный UDP-listener `127.0.0.1:9000` — сюда WG GoBackend пишет исходящие пакеты
- Делит N воркеров на **группы по 12** и каскадно стартует
- VK App ID `6287487` зашит в init() — это легитимный VK Desktop App

#### `dispatcher.go` — мультиплексор
- `readLoop`: один поток читает локальный UDP, кладёт пакет в `SendCh` следующего воркера по round-robin (если канал полный — пробует следующего, без блокировок)
- `writeLoop`: получает пакеты обратно по `ReturnCh` и пишет на адрес последнего отправителя
- `clientAddr` в `atomic.Pointer` — нет блокировок на горячем пути

#### `session.go` — одна TURN+DTLS сессия (сердце)
1. Dial TURN (TCP по умолчанию, опционально UDP) на `host:port` из VK-кредов
2. `pion/turn/v5` `NewClient` + `Listen()` + `Allocate()` → relay-адрес
3. `connutil.AsyncPacketPipe()` — двунаправленный pipe внутри процесса для DTLS-движка
4. Двусторонний прокси `relay ↔ pipeA` (TURN ChannelData ↔ raw UDP к peer)
5. `pion/dtls/v3` Client поверх `pipeB` — handshake к VPS через relay, **SNI = `calls.okcdn.ru`**, self-signed cert, **InsecureSkipVerify** (доверие через пароль на app-уровне)
6. Один воркер из группы делает `GETCONF:<localPort>|<deviceID>|<password>` → получает WG-конфиг
7. Параллельно: Writer (dispatcher → DTLS), Reader (DTLS → dispatcher), keepalive `WAKEUP` каждые 10 сек
8. Параллельный keepalive STUN binding каждые 10 сек чтобы NAT не закрыл relay

#### `group.go` — управление группами и **бесшовная ротация**
VK-credentials живут ограниченное время (`creds.Lifetime`, обычно ~часы). Просто рвать туннель плохо.
- Группа сначала **получает новые креды** (под `groupAuthMutex`), и только после этого убивает старый batch
- Каскад: следующая группа стартует через 2 сек после успеха предыдущей
- Адаптивный TTL: `Lifetime - 120s` берётся из ответа VK, не хардкод
- Stagger 500мс между воркерами внутри группы
- **Детекторы протухания кредов** (досрочная ротация по `refreshCh`):
  - 5+ воркеров получили `Quota Exceeded` (486)
  - 8+ воркеров получили `Unauthorized` / `Allocation Mismatch` / `error 29` / `error 508`
  - Мгновенно при `stream closed`
- Retry с jitter 5-15 сек, exit на FATAL_AUTH/мёртвый хеш

#### `protocol.go` — application-level протокол
```
client → server: "GETCONF:<localPort>|<deviceID>|<password>"
server → client: WG-config | "NOCONF" | "DENIED:wrong_password|expired|device_mismatch"
```
После — чистый passthrough WG UDP-фреймов в DTLS-payload.

#### `split_tunnel.go` — анти-петля
Если AllowedIPs = `0.0.0.0/0`, трафик до VK TURN-relay сам уходит в туннель → петля. Рекурсивная разбивка `0.0.0.0/0` минус CIDR-исключения:
- IP пира (VPS) — `/32`
- VK-сети: `95.163/16`, `87.240/16`, `93.186.224/20`, `185.32.248/22`, `185.29.130/24`, `217.20.144/20`
- RFC1918 (`10/8`, `172.16/12`, `192.168/16`)

### Маскировка от DPI

| Слой | Что видит наблюдатель |
|------|------------------------|
| L3-4 | TCP/UDP к публичному IP `*.okcdn.ru` или `vk-cdn.net` на портах 80/443/3478 |
| TLS-handshake | DTLS 1.2 с ClientHello SNI=`calls.okcdn.ru`, шифры — точь-в-точь VK звонок |
| Application | TURN ChannelData фреймы поверх DTLS — формат WebRTC-медиапотока |
| Behavior | Всплески исходящего/входящего, keepalive каждые 10 сек, ротация кредов раз в N часов — поведение реального голосового звонка |

DPI на основе сигнатур WireGuard (тип сообщения 0x01–0x04 в первых 4 байтах UDP) **не сработает**: WG-байты обёрнуты в DTLS Application Data record поверх TURN ChannelData.

### Серверная часть (VPS, `server.go`)
- Слушает `56000/udp` — DTLS endpoint
- После handshake: парсит GETCONF → выдаёт WG-конфиг (генерирует пиру IP в `10.66.66.0/24`)
- Парольная модель: главный пароль владельца + одноразовые ограниченные по сроку (через Telegram-бот в `botgen.go`)
- Пробрасывает WG payload в `wdtt0` интерфейс (внутренний WG на `56001/udp`) → NAT/MASQUERADE через `iptables` → выход
- `systemd`-сервис, деплоится по SSH из приложения

## Релиз 1.1.4 (2026-05-02) — слом архитектуры

> Сложно и важно. Если ставишь свежий релиз — реальная архитектура другая.

Из релиз-ноутов автора:
- **Полный отказ от WG + DTLS**, переход на **SOCKS5 поверх KCP + SMUX**
- Скорости 3-8 Мбит/с вместо 1-3, но потерян DTLS-камуфляж — "зона риска"
- Telegram-бот вырезан полностью
- Только ручная/Auto-WebView капча, RJS-режим выпилен
- Только 1 хеш (раньше до 3), потоки 1/9/18 (рекомендуется 18)
- Принимает ссылки `https://vk.com/call/join/*`, `vk.com/call/join/*`, `vk:<хеш>`
- На сервере улучшили работу с NFTABLES

Это уже не "обход DPI через маскировку под звонок", а скорее "прокси через VK-инфру". Сигнатура SOCKS5/KCP детектируется проще, чем DTLS.

Если нужна **скрытность** → откат на 1.0.6/1.1.0.
Если нужна **скорость** → 1.1.4.

## Установка (под актуальный 1.1.4)

### Что приготовить

**Телефон:**
- Android 10+ (SDK 29-36)
- ~100 МБ свободного места
- Разрешение установки из неизвестных источников

**VPS:**
- Любой провайдер с публичным IPv4
- 1 vCPU / 512 МБ RAM
- Свежее ядро: Debian 12 / Ubuntu 22.04+ / Rocky 9. **CentOS 7 не ставить**.
- root либо `sudo`-пользователь
- SSH-доступ по паролю (ключ скрипт деплоя пока не понимает)
- В firewall провайдера **открыть UDP 56000 и UDP 56001** на вход

**VK-аккаунт:**
- Лучше отдельный (не основной, не с привязкой к карте) — VK иногда банит за нестандартные сценарии звонков
- Нужна личная пустая группа

### Шаги

1. **APK на телефон**
   - Релиз: https://github.com/amurcanov/proxy-turn-vk-android/releases/tag/v1.1.4
   - `WDTT-arm64-v8a.apk` для современных телефонов, `universal` если не уверен
   - При установке Android спросит про неизвестный источник — разреши конкретно для браузера/файл-менеджера
   - Если Play Protect ругается — отключи проверку для WDTT

2. **VK-хеш звонка**
   - Создай **новую пустую группу** в VK (Сообщества → Создать)
   - Заходи в группу → Звонки/Видеозвонок → начни групповой звонок
   - В окне звонка: «**Скопировать ссылку приглашения**» → получишь `https://vk.com/call/join/<hash>`
   - **Не закрывай звонок!** Сверни VK. Хеш живёт пока комната активна.
   - При выходе из звонка тапай **«Просто завершить»**, НЕ «Завершить для всех» (иначе хеш мёртв)

3. **Деплой**
   Открой WDTT → вкладка **«Деплой»**:
   - Хост: IP/домен VPS
   - Логин: `root` или sudo-юзер
   - Пароль: SSH-пароль
   - SSH-порт: 22 (или нестандартный)

   Вкладка **«Секреты»**:
   - **Пароль туннеля** — длинный, запиши (нужен на стороне клиента для подключения)
   - Telegram-поля в 1.1.4 не используются

   Возвращайся в «Деплой» → **«Установить»**. 2-5 минут:
   - Скачивается `wdtt-server` бинарь
   - Создаётся `systemd`-юнит `wdtt.service`
   - Настраивается NAT (NFTABLES/iptables), WG-интерфейс `wdtt0`, открываются порты
   - В логах: `[✓] Деплой завершён`

4. **Подключение**
   Вкладка **«Туннель»**:
   - Сервер: IP VPS
   - VK-хеш: ссылка целиком или сам хеш
   - Пароль: пароль туннеля из «Секретов»
   - Потоков: **18** (или 9 если 18 нестабильно)

   Жми **«Подключить»** → Android попросит разрешение VPN → дай.

   **Первое подключение почти всегда требует капчу:**
   - Auto-WBV попробует сам
   - Не справился → откроется WebView со слайдером, реши вручную
   - Дальше до ротации кредов не спросит

5. **Исключения приложений** (опционально)
   Вкладка **«Исключ.»**:
   - **ЧС**: отмеченные **мимо** туннеля (банковские, такси, локальные мессенджеры)
   - **БС**: только отмеченные **через** туннель
   - WDTT и VK-клиенты исключаются автоматом (иначе TURN сам уходит в туннель = петля)

### Скорость
- 1.0.6/1.1.0: ~1-3 Мбит/с
- 1.1.4: ~3-8 Мбит/с
- Потолок про сёрфинг и YouTube 480p-720p. 4K/большие загрузки — нет. Игры — нет (latency)
- Зависит от: страны VPS, нагрузки на VK TURN, времени суток, качества мобильной сети

## ⚠️ Российский VPS — отдельные соображения

Если VPS физически в РФ:

**Плюсы:**
- Latency до VK TURN минимальный (single-digit ms)
- Стабильность канала между VPS и VK-инфрой выше

**Минусы / парадоксы:**
- Трафик **от VPS в "запрещённые" ресурсы** (YouTube/Twitter/etc) попадает под российский DPI на исходящем — то есть YouTube будет так же замедлен, как и без VPN
- Российские VPS-провайдеры подчиняются НСС/блокировкам по линии РКН — то, что заблокировано для конечного пользователя, **скорее всего заблокировано и для VPS**
- VK TURN видит исходящий трафик от российского IP — никаких "иностранных VPN" не маскируется

**Когда туннель с РФ VPS осмыслен:**
1. Обход локальных DPI: корпоративная/университетская сеть, публичный Wi-Fi с фильтром, мобильный оператор с блокировкой Discord/etc
2. Обход блокировок мессенджеров (если конкретный мессенджер блокируется на уровне оператора, но доступен с любого "обычного" IP)
3. Тестирование — посмотреть, как сайт ведёт себя с другого IP
4. Динамические блокировки на стороне сервиса (anti-fraud) — сменить IP

**Когда НЕ имеет смысла:**
- Для YouTube/Twitter/любых ресурсов, которые блокируются/замедляются на уровне магистрали РФ
- Для скрытия от российского ГосСОПКА/СОРМ — ровно ничего не скрывается, наоборот, добавляется лог-цепочка

**Возможные направления:**
- Если хочешь обходить именно российские блокировки на YouTube/etc — нужен **зарубежный** VPS
- Если хочешь обойти именно DPI на корпоративе/Wi-Fi/мобильном — РФ VPS подойдёт, но проще VLESS/Reality напрямую
- Если хочешь именно WDTT-схему и зарубежный VPS — учти, что скорость будет ниже (3-8 Мбит/с верхняя граница превратится в ~1-3)

## Если что-то ломается

| Симптом | Что делать |
|---------|-----------|
| Деплой падает на NAT/firewall | Проверь UDP 56000/56001 в панели VPS |
| Подключение зависает на «Получение кредов» | VK закрыл звонок / хеш мёртв — пересоздай |
| Бесконечная капча | Используй WebView-режим (ручной) во вкладке «Тоннель» → настройки капчи |
| Скорость 0 / нет трафика | UDP заблокирован у провайдера VPS — попробуй другой VPS / TCP-режим |
| VPN-разрешение не появляется | Уже работает другой VPN — отключи в системных настройках Android |
| Краш на запуске | «Информация» → «Собрать отчёт» → откати на v1.0.6 |

Логи туннеля: вкладка **«Логи»** в приложении.

## Слабые места WDTT в целом

- **Зависимость от VK**: достаточно VK закрыть API публичных групповых звонков или ввести WAF на TURN — схема ляжет одномоментно
- **TURN-квоты**: VK может тротлить allocation-ы → нужна ротация хешей
- **Капча**: ключевая точка отказа на bootstrap, ручное решение убивает UX
- `vkAppID/Secret` хардкоженные → если VK заблокирует именно этот app — все клиенты разом мёртвые
- Self-signed сертификат + InsecureSkipVerify → MITM возможен между клиентом и VPS на уровне `pipeA ↔ relay ↔ peer` (хотя пароль это ловит)
- WireGuard-протокол сам по себе детектируется по handshake-сигнатуре, поэтому **планируется миграция на AmneziaWG** в 1.5-2.0
- 1.1.4 потерял DTLS-камуфляж → SOCKS5+KCP детектируется проще

## Сравнение с альтернативами

| Решение | Скорость | DPI-обход | Сложность | Зависимость |
|---------|----------|-----------|-----------|-------------|
| WDTT 1.0.6 (WG+DTLS) | 1-3 Мбит | ⭐⭐⭐⭐⭐ (под VK call) | средняя | VK инфра |
| WDTT 1.1.4 (SOCKS5+KCP) | 3-8 Мбит | ⭐⭐⭐ | средняя | VK инфра |
| VLESS+Reality | 50-200 Мбит | ⭐⭐⭐⭐⭐ (под TLS) | высокая | свой VPS |
| Shadowsocks/v2ray | 30-100 Мбит | ⭐⭐⭐ | средняя | свой VPS |
| AmneziaVPN | 20-100 Мбит | ⭐⭐⭐⭐ | низкая | свой VPS |
| Outline | 50-200 Мбит | ⭐⭐ | низкая | свой VPS |

WDTT нишевый: специфический сценарий, когда **именно** VK-маскировка нужна (например, в сети заблокированы все TLS-сигнатуры VPN, но VK-звонки работают). В большинстве случаев VLESS+Reality эффективнее по всем параметрам.
